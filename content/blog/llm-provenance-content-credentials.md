Legal asked a straightforward question during a pilot: when the marketing agent publishes an image, can we show **who** generated it, **with which model**, and **whether a human approved** it before release? Screenshots of chat logs do not survive a dispute. EXIF fields are trivially stripped. A PDF "Generated by AI" footer is not cryptographically verifiable.

Content Credentials—standardized by the Coalition for Content Provenance and Authenticity (C2PA)—attach a signed manifest to an asset describing its history: creation tools, edits, generative steps, and optional reviewer assertions. For agent products that emit images, audio, video, or even structured exports, provenance is becoming a ship requirement, not a research demo. This article walks through what to sign, where to sign it, and where the trust model stops.

## Manifests, not magic

A Content Credential is a **C2PA manifest** stored in JUMBF format, embedded in the file (JPEG, PNG, WebP, PDF, MP4) or carried as a sidecar. The manifest lists **assertions**—structured claims like:

- `c2pa.actions`: ordered list (`c2pa.created`, `c2pa.edited`, `c2pa.ai_generative_training` opt-out flags where applicable)
- `c2pa.ingredients`: parent assets if the output composes earlier files
- Custom assertions: `agent.session_id`, `model_id`, `prompt_hash` (careful with PII)

Each manifest is signed. Verifiers walk the certificate chain to a trust anchor (your org cert or a public CA participating in the C2PA trust list).

What verification **does** return:

- Manifest intact (not tampered since signing)
- Signer identity bound to certificate
- Action history as claimed by signers

What it **does not** return:

- Factual accuracy of the content
- Proof the model wasn't jailbroken
- Guarantee the file you hold matches what was signed if someone stripped metadata offline

Set expectations with product and legal teams early.

## Architecture for agent output signing

```
Agent worker completes render
        │
        ▼
Bytes buffer (PNG/MP3/PDF)
        │
        ▼
Provenance service ──► Build manifest (actions + custom assertions)
        │
        ▼
KMS sign manifest ──► Embed JUMBF in asset
        │
        ▼
Object storage + CDN ──► Optional public verify endpoint
```

Keep signing out of the LLM loop. Tool workers produce bytes; a **provenance microservice** owns certificates, assertion schemas, and rate limits. Agents call it with a stable `content_id` and metadata; the service returns signed bytes or a manifest blob to embed.

```typescript
// Agent tool handler — delegate signing, never hold private keys
export async function exportGeneratedImage(
  session: AgentSession,
  imageBuffer: Buffer,
  meta: { modelId: string; promptHash: string; toolRunId: string },
): Promise<SignedAsset> {
  const response = await provenanceClient.sign({
    assetType: "image/png",
    payloadBase64: imageBuffer.toString("base64"),
    assertions: {
      actions: [
        { action: "c2pa.created", softwareAgent: "AcmeAgent/2.4" },
        { action: "c2pa.ai_generative", model: meta.modelId },
      ],
      custom: {
        "acme.agent.session_id": session.id,
        "acme.agent.prompt_hash": meta.promptHash,
        "acme.agent.tool_run_id": meta.toolRunId,
      },
    },
  });
  return {
    bytes: Buffer.from(response.signedPayloadBase64, "base64"),
    manifestId: response.manifestId,
    verifyUrl: `https://app.example.com/verify/${response.manifestId}`,
  };
}
```

Hash prompts, do not embed raw prompts, unless policy explicitly allows and GDPR/CCPA review covers it.

## Human-in-the-loop as a second assertion

Fully automated signing proves the agent produced the file. Regulated workflows often need **approval** as a distinct step:

1. Agent signs with `c2pa.created` + generative metadata.
2. Asset enters review queue.
3. Reviewer approves; approval service adds manifest **ingredient** linking prior manifest and signs `c2pa.published` or custom `acme.review.approved` with reviewer credential.

Downstream verifiers see a chain. Rejections should not publish signed assets to public buckets—unsigned drafts stay in private storage.

## Implementation notes by format

**Images (PNG/JPEG):** Embedding survives moderate recompression better than invisible watermarks alone, but aggressive social platform transforms still kill JUMBF. Test export paths from your CDN.

**PDF:** Useful for agent-generated reports. Sign after final pagination; late edits invalidate manifest.

**Audio/video:** Latency-sensitive; signing service should stream where SDK supports it. Agent podcasts and voice replies are emerging disclosure targets in EU and US state guidelines.

**Text-only outputs:** C2PA targets binary assets. For markdown or JSON, pair with detached manifest sidecars or move to signed PDF exports for externally shared deliverables.

## Verification UX matters

Engineering can embed credentials perfectly; users still need understandable UI:

- **Green check**: manifest valid, signer recognized.
- **Yellow**: manifest present but signer unknown or certificate expired.
- **Gray**: no manifest—legacy upload or stripped metadata.

Link to a detail drawer listing actions in plain language, not OID strings.

```python
# Server-side verification sketch using c2pa-python
from c2pa import Reader

def verify_upload(data: bytes) -> dict:
    reader = Reader(data)
    status = reader.validate()
    return {
        "valid": status.is_valid,
        "signer": status.signer_info.common_name if status.signer_info else None,
        "actions": [a.action for a in reader.manifest.actions],
        "warnings": list(status.warnings),
    }
```

Log verification results for compliance analytics; do not log full file bytes.

## Trust limits and threat model

Attackers can:

- Strip manifests and re-share (your UI should detect absence where expected)
- Sign fake manifests with self-signed certs (verification must anchor to trust list)
- Generate deepfakes outside your product without any credential

Credentials raise the cost of **passing off third-party fakes as your platform's output**. They do not stop all synthetic media on the internet.

Inside your boundary, protect signing keys aggressively: KMS, short-lived tokens to call provenance service, mTLS from agent workers, audit every sign request with `tenant_id` and `session_id`.

## Policy alignment

EU AI Act transparency duties, US state deepfake laws, and platform partner programs increasingly ask for **machine-readable disclosure**. Content Credentials are one interoperable format—not the only one. Map legal phrases ("synthetically generated") to concrete manifest actions your lawyers accept.

Maintain a **manifest schema version** internally. When you add assertions, bump version and keep verifiers backward compatible for older assets stored in customer libraries.

## Operational checklist

- [ ] Central signing service with KMS-backed keys  
- [ ] Prompt/content hashes instead of raw text in manifests  
- [ ] Separate automated vs human-approved signing profiles  
- [ ] CDN tests confirming embed survival  
- [ ] Public verify page + API for enterprise customers  
- [ ] Runbook for certificate rotation without invalidating historical assets (timestamp authority + archived certs)

Provenance Content Credentials turn agent outputs from ephemeral chat attachments into auditable artifacts—if you invest in signing infrastructure, verification UX, and honest communication about limits.

## Multi-tenant and white-label considerations

SaaS agent platforms sign on behalf of customers publishing to their own channels. Decide upfront:

**Platform signer only.** Simpler trust chain; customers rely on your org certificate. Suitable when you host all exports.

**Customer sub-certificates.** Enterprise tenants bring C2PA-compatible certs; your service signs as delegated issuer. Needed when their legal team requires their name on the manifest, not yours.

**Tenant isolation in audit logs.** Every sign request logs `tenant_id`, `user_id`, `content_type`, and manifest hash—never store raw generative prompts in the same table if retention policies differ.

White-label mobile apps may need distinct trust anchors per app bundle. Maintain a mapping from `app_id` to certificate profile in the provenance service rather than hardcoding one global signer.

## Redaction and takedown workflows

When users delete content, credentials embedded in copies already downloaded cannot be recalled—similar to PDF redaction limits. Your product obligations:

- Stop serving signed bytes from managed storage on delete.
- Invalidate verify URLs that cached manifest metadata server-side.
- Document in ToS that exported files may retain credentials client-side.

For moderation takedowns, add manifest actions where policy allows (`c2pa.redacted`) rather than silently swapping unsigned files—downstream enterprise verifiers prefer explicit redaction events over mysterious metadata loss.

## Interop with platform export APIs

Some creative tools now accept C2PA on import. Test round trips: agent export → desktop editor → re-export. Manifests may append ingredient links rather than replace—verify your signing service handles **manifest chaining** when users edit signed assets in external tools and re-upload.

```typescript
// When re-ingesting a user-edited signed asset
const existing = await provenanceClient.inspect(uploadBuffer);
await provenanceClient.sign({
  assetType: "image/png",
  payloadBase64: uploadBuffer.toString("base64"),
  ingredients: [{ manifestId: existing.manifestId, relationship: "parentOf" }],
  assertions: {
    actions: [{ action: "c2pa.edited", softwareAgent: "UserEditor/unknown" }],
  },
});
```

Chain integrity is where provenance adds value over static watermarks—you preserve history instead of overwriting it.
